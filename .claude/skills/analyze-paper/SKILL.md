# analyze-paper

Produce a **deep, structured analysis of a paper by actually reading its full
body** — not the arXiv abstract — and save it into the paper's KB record so it is
visible in the app and (once embedded) searchable via `kb_search`.

Background: in this KB, the stored "abstract" is just the arXiv `<summary>` copied
verbatim at ingest (`src/ingest/arxiv.rs`). It is *not* LLM-generated. This skill
exists because a full agentic read produces something substantially more useful
than that blurb. The closest existing artifact is the on-demand "Clean Read"
(`reader.md`), which is a distillation, not an analysis.

## Arguments

- `/analyze-paper <arxiv_id>` — analyze one paper (e.g. `2504.19874`).
- `/analyze-paper all` — sweep every paper in the library.
- `/analyze-paper` (no arg) — ask which paper, or offer `all`.

## Where things live

- **Library root = `$KB_ROOT`. Resolve it FIRST — do not assume `~/arxiv-kb`.**
  The repo exports it in `/Volumes/x/kb/.env.local.sh`
  (`export KB_ROOT=/Volumes/x/arxiv-kb`). Read that file (or `echo $KB_ROOT` if
  already exported). A stale copy may exist at `~/arxiv-kb` with far fewer papers
  — use the `$KB_ROOT` value. (The MCP tools resolve their own root, but reads of
  `sections.md`/`metadata.json` below are direct file reads, so get this right.)
- Per paper: `$KB_ROOT/<id>/` containing `metadata.json` (has the `abstract`,
  `title`, `authors`), `sections.md` (extracted + classified body — **read this**),
  and `paper.pdf` (fallback if `sections.md` is missing/empty).
- The `arxiv-kb` MCP server must be connected. Tools used: `kb_get_paper` (read
  metadata + existing notes), `kb_add_note` (append the analysis to `notes.md`).

## Procedure (per paper)

1. **Resolve the paper.** First resolve `$KB_ROOT` (see "Where things live"). If
   given an id, confirm `$KB_ROOT/<id>/` exists. For `all`, list directories under
   `$KB_ROOT/` that contain a `metadata.json` (skip `.arxiv-kb/`). Non-arXiv docs
   (e.g. web captures) are fine too.
2. **Read the FULL body.** Read `$KB_ROOT/<id>/sections.md` in its entirety
   (it can be large — page through it; do not analyze from the first page alone).
   If absent/empty, read `paper.pdf` directly. Also read `metadata.json` for the
   title and the existing `abstract` so the analysis can go *beyond* it.
3. **Check for an existing analysis** via `kb_get_paper` — if `notes.md` already
   contains a "Claude Code analysis" block, tell the user and ask whether to
   regenerate (append a fresh one) or skip.
4. **Write the analysis** following the structure below.
5. **Save it** with `kb_add_note(paper_id, note)`. The note re-embeds
   automatically *iff* `OPENAI_API_KEY` is valid — see Embedding note.
6. For `all`, do them sequentially and end with a one-line summary
   (analyzed / skipped / failed counts).

## Analysis structure (keep these sections, adapt depth to the paper)

Lead the saved note with a `## Claude Code analysis — <short title>` heading, then:

- **One-line** — the thesis in one sentence (mechanism + why it matters).
- **Core mechanism / trick** — how it actually works, numbered steps. The part the
  abstract omits.
- **Key design insight** — the non-obvious decision or distinction that makes the
  paper matter (e.g. two objectives that look similar but aren't).
- **Guarantees / headline numbers** — concrete bounds, rates, benchmark deltas.
  Quote real numbers from the paper, not vague claims.
- **Empirical takeaways relevant to KB** — explicitly connect to this project's
  own setup where applicable: 1536-d embeddings (`text-embedding-3-small`),
  TurboVec / TurboQuant quantization, KV-cache, retrieval/ranking. This is what
  makes the analysis worth more than the abstract.
- **Limitations / caveats not in the abstract** — assumptions, weak regimes,
  things the authors downplay.
- **So what (for KB / TurboVec)** — the single most actionable implication for
  this codebase, if any.
- End with: `— Generated by Claude Code from a full read of the paper body.`

Be specific and grounded in the text. No filler. If the paper has no KB-relevant
angle, say so briefly rather than inventing one.

## Embedding note (always surface this)

`kb_add_note` returns whether re-embedding succeeded. If it reports the note was
saved but re-embedding was **deferred** (commonly `invalid OPENAI_API_KEY`/HTTP
401), tell the user: the analysis is persisted to `notes.md` and visible in the
app, but **not yet searchable** until embeddings run. To fix: export a valid
`OPENAI_API_KEY` and run `kb reindex` (or let a running `kb watch` pick it up).
Do not treat a deferred embed as a failure — the analysis is saved.
